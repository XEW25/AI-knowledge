## 核心问题(两条)

1. **端侧资源受限(单机层面)**:实时推理与持续学习**争夺同一份算力**,无法兼得。
2. **场景高度个性化、繁多且各端异构(系统层面)**:云-端协同进化需同时兼顾**本地特化**与**车队共智**,难以两全。

**重定位(本框架的立论)**:核心本质**不是"怎么学"**(学习算法 ML 早有答案),**而是"怎么让学习在具身部署的现实里活下来"**——扛得住资源、实时安全、间歇连网、车队异构,以及解耦本身带来的协同难题。

> 这两条恰是**具身特有**的:纯云 ML 系统两者都没有(资源充裕、单一模型无众多物理场景)。而且分属**单机**与**系统**两个层级。安全/可控是**贯穿两条核心的护栏**,不单列为第三核心。

## 框架总览:一座对称桥 + 两个不对称引擎

- **边端引擎**:受限 / 实时 / 作动安全 / 专家**消费者**。
- **云端引擎**:算力充裕 / 离线 / 车队视角 / 专家**生产者**。
- **桥(对称,两端)**:让两个**不对称**引擎协同而不必同构。

**"谁能做什么"判据**:凡需"**全车队视角**"(聚合、跨场景泛化、产新技能、权威版本)→ 只能在云;凡需"**实时 + 本地**"(快适应、作动安全、断网自治)→ 只能在端。比按"大模型/小模型"分更本质。

**关键架构选择 = B(模块化独立专家)**:端侧专家是按场景子任务/子技能各自训练的独立模块;云脑(VLM + 世界模型)推理后下发**结构化高层指令**(子任务序列 + 行为约束 + 阶段验证标准)。详见 [[Embodied Brain Models]] 的 VLM-as-actor/encoder 与两层耦合框架。

## 边端:三类职责(学得好 / 稳 / 协同)+ 端口

| 类别 | 职责 | 关键内容 | 对外端口 |
|---|---|---|---|
| **① 脑启发演进算法** | 学得好(受限端侧高效学)| CLS 快/慢双速率、STDP/R-STDP、神经调质门控、元/结构可塑性;异构算力/DSA 编排 | **异构硬件 HAL 口**;运行时**经验/信号 in 口**(验证结果+前向误差+接管)|
| **② 系统安全策略** | 学得稳(可控不跑偏)| 信号完整性、受约束更新、影子/金丝雀/回滚、Simplex 运行时保障、不学习的经典地板 | **redeploy 管线口**(推理引擎模块);**云审批口** |
| **③ 端云协同组件** | 学得协同(接入闭环)| 能力画像同步、接口契约共版本化、能力缺口信号 | **网络通信口**(异步/store-and-forward/QoS);本地编排器口 |

> 形态上是**六边形/端口-适配器架构**:三类纯演进逻辑 + 四个适配器口(HAL / redeploy / 网络 / 运行时)。好处:可脱离具体硬件与云端实现被**单独讲清、单独验证**;四个口也恰好对应四个团队/供应商边界。

## 云端:四类职责

| 类别 | 职责 | 关键内容 |
|---|---|---|
| **云① 持续学习引擎** | 大脑(世界模型+推理+规划)从车队经验持续变强 | 大规模(再)训练、车队级 RL(π0.6 Recap 放大版)、大规模持续学习/防遗忘。**与边端不同:用"规模"而非"脑启发省力"** |
| **云② 车队学习与聚合** | 多端/多场景经验综合(端做不到、云独有)| **adapter 级联邦聚合** + 隐私过滤经验融合喂大脑 + 跨场景泛化 + 个性化-vs-泛化(模块粒度)+ 跨本体联邦蒸馏 |
| **云③ 技能工厂与能力治理** | 产/验/版本化/退役专家 + 权威能力画像 | **新专家训练 = 模仿优先(teleop 接管为主要示范源,DAgger)+ 数据稀缺路线可配置 + RL 做 refinement**;权威能力登记表 |
| **云④ 端云协同组件(云端)** | 桥的云端(对称于边③)| 上行摄取/去重、分阶段车队 rollout、契约共版本化、**下发前车队级验证门**、签名/防投毒 |

> **不对称(汇报里要点明)**:边②的"实时/作动安全"在云无直接镜像(云不碰作动器→云"安全"=车队治理,折进云③/④);"技能工厂"是云独有(边只消费、云生产)。**只有"桥"是对称的。**

### 演进接口(B 之下,大脑↔专家传什么)
大脑演进后**不需要推权重给专家**——更好的指令经**运行时通道**自然下发。演进平面只需显式同步三样:**① 能力画像**(专家变了→更新云对它的能力模型,而非传权重);**② 接口契约共版本化**(子任务/约束/验证"语言"两侧对齐);**③ 能力缺口信号**(大脑要的技能没有→催生新专家)。

## 四个关键技术(2+2 矩阵)

| | 直接解核心 | 让它跑起来的使能 |
|---|---|---|
| **资源轴(核心①)** | **T1 参数高效+错峰的端侧自演进**(LoRA/adapter + CLS 快慢 + DMD 步数蒸馏 + 异构算力/DSA)| **T2 可控自演进 / 运行时安全**(Simplex + 影子/金丝雀/回滚 + 经典地板)|
| **协同轴(核心②)** | **T3 模块化联邦协同进化**(按技能 adapter 级 FL + 个性化-vs-泛化模块化共享 + 缺口触发技能生产)| **T4 端云协同接口**(能力画像 + 契约共版本化 + 缺口信号)|

**抽掉任一即塌**:无 T1 端侧学不动;无 T2 不敢部署;无 T3 场景成孤岛(无"协同");无 T4 双方"语言"漂移、飞轮断。
**成熟度/创新点**:T2(Simplex)主要从航空/自动驾驶借;T1 的 LoRA、T3 的 FL 框架成熟;**创新密度集中在 T1 的端侧 RL+脑启发+DMD、T3 的具身模块化联邦、T4 的"给学习出来的接口做共版本化"**。

## 核心① 的证据(端侧资源)

**机制(硬)**:训练显存 ≈ **8×** 推理(混合精度账:权重+梯度+Adam 2×+激活)。
**硬件(已核实,见 figures 笔记表)**:Thor headline **2070 是稀疏 FP4**(纯推理),训练相关 **FP16 ~258 稠密**(差 ~8×);Orin INT8 275 vs FP32 5.3(~52×);带宽 Thor **273 GB/s** vs H100 3350 / B200 8000(少 ~12–29×);**算力 Orin→Thor 涨 7.5×、带宽仅 1.33×**。
**业界定位(已核实)**:NVIDIA "三计算机"——DGX 训练 / Omniverse-Cosmos 仿真 / **Jetson Thor 推理**;GR00T 工作流=**服务器微调→Thor 部署**。即端侧训练非既有实践。
**旁证**:TinyML 训练墙(On-Device Training Under 256KB)、LoRA(全量微调太重才有 PEFT)。

> ⚠️ **诚实修正(算过才知道)**:对 **2B 专家 + 128GB Thor**,显存其实**够用**(全量微调一个 2B ≈ 65GB < 128GB)。**真正的瓶颈不是显存容量,而是:实时算力争用(训练抢 GPU/带宽→推理掉帧/丢标准)+ 功耗(持续满载)+ 吞吐(慢、单机不可扩展)。** 故核心① 的主图用**"实时算力争用→推理性能下降"**,显存溢出仅在"大模型/机载大脑/多专家同时全量训"时成立(降级为附注)。

**配图**:实时算力争用→延迟越过实时标准→错峰巩固(见 figures 笔记)。

## 核心② 的证据(跨场景无协同)

**结构**:孤岛(节点无连线=无协同,各自微调/RL)vs 协同(经云中枢经验上行+共享下发)。
**标度(概念示意,非实测)**:孤岛成本 ≈ N×单场景(线性);协同 ≈ 共享基座 + 小 adapter(亚线性)。诚实保留**交叉点**(场景少时孤岛够用、规模化后协同制胜)。亚线性来自 **B 模块化 + 云② 联邦**(通用技能共享、特化本地)。

**配图**:孤岛↔协同拓扑、成本标度曲线(见 figures 笔记)。

## Open Problems(汇报里单独标)

- **开放世界技能的上线前验证(覆盖性)**——② 唯一的硬骨头:回归测试盖不住长尾,影子只覆盖遇到过的场景。
- **个性化 vs 车队泛化**——模块粒度共享是钥匙,但取舍未定。
- **隐私保护下仍"有用"的经验上行**(家庭/工厂数据)。
- **"学习出来的接口"如何协同演进**(契约本身在变,比 API 版本化难)。
- **在线适应**(STDP/R-STDP)在机器人技能尺度**未被证明**;可靠路径仍是离线 replay+RL 巩固。
- **边缘世界模型可行性**(Kairos 式像素级世界模型能否端侧实时闭环)。
- FP4 headline 与训练精度的鸿沟在新硬件上**反而更宽**。
